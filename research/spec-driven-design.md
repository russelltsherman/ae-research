# spec driven design

> Generated by /batch-research → /deep-research on 2026-06-24

## Summary
Spec-Driven Development (SDD) is a software methodology in which a precise specification—rather than code—is the primary source of truth, with code treated as a generated or verified secondary artifact ("the spec declares intent; the code realizes it"). Though spec-first thinking is not new (it has lineage in TDD, BDD, and formal methods), the 2024-2026 resurgence was driven specifically by AI coding assistants, which are strong at pattern completion but poor at inferring intent, making explicit specs valuable. In practice, SDD is operationalized by tools such as GitHub Spec Kit (a Spec -> Plan -> Tasks -> Implement workflow driven by slash commands, each phase producing a Markdown artifact that feeds the next) and Amazon Kiro (a VS Code/Code-OSS fork that turns natural-language requirements into requirements, design, and task artifacts before implementation). Practitioners and analysts describe a spectrum of rigor—spec-first, spec-anchored, and spec-as-source—and an active debate between "radicals" (spec is the sole source of truth, code is a byproduct) and "traditionalists" (executable code remains authoritative). The main acknowledged tradeoffs and failure modes are spec drift and hallucination, which are inherently hard to avoid and demand deterministic CI/CD safeguards, plus criticism over spec overhead and elaborate, opinionated workflows.

## Caveats
Source mix: definitional and workflow claims rest on strong primary sources (arXiv 2602.00180, GitHub spec-kit repo/docs/blog, Kiro docs). However, much of this primary material is vendor/promotional (GitHub, AWS, IBM) describing each tool's stated intent and design, not independently validated empirical outcomes — no claim here demonstrates SDD measurably improves delivery speed or quality. Several claims are descriptive/definitional ('SDD treats specs as...', 'is positioned as...') rather than efficacy claims, so they survive verification precisely because they assert framing, not proven results. The 'executable specification' language is metaphorical and AI-mediated; specs do not literally self-execute. The criticism/failure-mode findings (debate, spec drift, overhead) rest largely on single secondary or blog-quality sources (Thoughtworks, Marmelab, Scott Logic) and are thus medium confidence. The three-tier rigor taxonomy originates with Thoughtworks/Birgitta Bockeler, adopted by IBM and the arXiv paper. Time-sensitivity is significant: this is a fast-moving 2024-2026 wave; tool command sets (e.g., spec-kit slash commands), Kiro's status, and Thoughtworks Technology Radar placement can change. Three related claims were refuted during verification (notably an IBM-attributed 'detailed implementation details agreed before development' definition, 0-3), so avoid over-precise definitional framings. The arXiv paper is a single-author preprint, not peer-reviewed. Notable absence of coverage in surviving claims: the question asked about OpenAPI/contract-first/API-first design and TLA+/formal specifications as intellectual lineage and tooling, but no verified claim substantiates these specifically — they appear only in passing verifier context.
