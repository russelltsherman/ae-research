# spec engineering

> Generated by /ae-research → research on 2026-06-25

## Summary
"Spec engineering" today spans two largely distinct traditions. The dominant, AI-native one is Spec-Driven Development (SDD), popularized by GitHub's open-source Spec Kit toolkit, which inverts the conventional relationship between code and specs: the specification becomes the primary, executable artifact and code becomes a regenerated downstream expression, with AI agents generating, testing, and validating code from a spec treated as the source of truth. SDD is operationalized through structured, multi-phase workflows (Specify -> Plan -> Tasks -> Implement, surfaced as slash commands) that separate the "what" from the "how," and frameworks like BMAD-METHOD similarly front-load PRDs/briefs/UX specs in an upfront planning phase. The older tradition is formal/requirements specification — the classical SRS discipline codified by IEEE 830 (eight quality attributes such as correct, unambiguous, complete, testable, traceable) and formal specification languages like TLA+ and Alloy with their verification toolchains (TLC, Apalache, TLAPS for TLA+; bounded SAT-based analysis for Alloy). Notably, industrial formal-spec practice remains largely confined to early design (81% of reviewed TLA+ cases) rather than synchronized with production code, and SDD's strongest "spec-as-source" claims (code never manually edited; the spec-implementation gap eliminated) are framework aspirations whose real-world efficacy is contested.

## Caveats
Source quality and time-sensitivity vary by finding. (1) AI-native SDD claims rest heavily on vendor/advocacy sources (GitHub's spec-kit repo, GitHub blog, Microsoft developer blog). These are accurate as descriptions of how the frameworks position themselves, but the strong claims — that the spec-implementation gap is eliminated, that code is 'never manually modified,' that specs 'directly generate' implementations — are framework aspirations, not independently verified outcomes. In practice AI agents (not specs) generate code, and verifiers flagged the efficacy debate (Fowler, Thoughtworks, dev.to critics) over whether SDD actually works as claimed. A related claim that human-refined specs reduce LLM code errors by up to 50% was REFUTED (0-3). (2) SDD is an emerging 2024-2026 methodology; tooling details drift fast (e.g., Spec Kit now uses namespaced /speckit.* commands; a claimed 7-step lifecycle was refuted, as was a fixed six-section IEEE 830 template). (3) The formal-methods comparison (Alloy vs TLA+) draws on a single 2016 paper (arXiv 1603.03599) that predates Alloy 6 (2021); the static/dynamic framing remains directionally correct per community sources but Alloy 6 narrowed the temporal gap — two of those claims were 2-1 split votes (medium confidence). (4) IEEE 830-1998 is superseded by ISO/IEC/IEEE 29148:2011; the eight quality attributes remain canonical but the standard itself is dated. (5) Several live source URLs (IEEE Xplore, Oregon State textbook) returned 403/empty on direct fetch; those claims rest on search-surfaced abstracts plus independent corroboration rather than direct page reads.

## Open questions
- Does SDD measurably improve software quality, delivery speed, or defect rates versus traditional or 'vibe coding' approaches? The one efficacy claim (50% error reduction) was refuted, and no verified controlled-study evidence survived — the case for SDD remains conceptual/advocacy-based.
- How do the AI-native SDD tradition (Spec Kit, BMAD) and the formal-specification tradition (IEEE 830 SRS, TLA+, Alloy) relate or interoperate, if at all? The sources treat them as separate worlds; whether formal verification techniques are being applied to AI-generated specs is unexplored.
- Given that industrial TLA+ is overwhelmingly early-design and spec-to-code synchronization is rare (19%), what tooling or methods are emerging to keep formal specs in sync with evolving production code — the same gap SDD claims to solve via regeneration?
- How stable is Spec Kit's workflow and command surface, and what is its actual production adoption versus GitHub star count — i.e., is SDD used at scale or still largely experimental as of 2026?
