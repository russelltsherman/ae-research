# Document-Driven Design

> Generated by /batch-research → /deep-research on 2026-06-24

## Summary
Document-Driven Design/Development is a family of software engineering methodologies in which written artifacts — design docs, specs, RFCs, ADRs, and PRD-style narratives — are authored before and to drive implementation, on the premise that prose is a better tool than code for solving problems early and that an undocumented feature effectively does not exist. The approach spans several named variants: Documentation-Driven Development (write user-facing docs first, then TDD to match), Spec-Driven Development (the spec is the source of truth and code is merely its expression), and decision/proposal records (ADRs, RFCs, RFDs, design docs). It has deep historical roots — the RFC format dates to Steve Crocker and ARPANET in 1969, deliberately named to invite discussion rather than declare — and is widely adopted at scale: Google's informal design docs, Amazon's 2004 narrative-memo mandate and PR-FAQ, Rust's RFC process, and Oxide's RFDs. Benefits center on cross-team transparency, organizational memory, and catching architectural problems before production; the principal criticism, inherited from the Big Design Up Front debate, is poor adaptability to change and analysis paralysis when planning cost exceeds the cost of fixing. The evidence base is strong: most definitional and historical claims are anchored in authoritative primary sources with unanimous adversarial votes.

## Findings
- **Documentation-Driven Development (DDD) is defined by the principle that, from a user's perspective, an undocumented feature does not exist and an incorrectly documented one is broken; it prescribes writing user-facing documentation first, then following a defined workflow (write docs, gather feedback, do TDD aligned with docs, push to staging, functional test, ship, publish docs, increment versions).** — confidence: high
  - Evidence: The canonical zsup gist (2014) states verbatim: 'from the perspective of a user, if a feature is not documented, then it doesn't exist, and if a feature is documented incorrectly, then it's broken.' It mandates documenting the feature first and reviewing docs with users before any development begins, then specifies the 8-step workflow (write docs -> get feedback -> TDD aligned with docs -> push to staging -> functional testing -> deliver -> publish docs -> increment versions). Multiple independent secondary sources cite this exact gist as the definition. Note: 'DDD' more commonly denotes Domain-Driven Design; this finding is scoped to Documentation-Driven Development specifically.
  - Sources: https://gist.github.com/zsup/9434452
- **Spec-Driven Development (SDD) inverts the traditional spec/code relationship: the specification becomes the primary artifact and source of truth that generates working systems, while code is merely its expression in a particular language/framework — requiring specs precise and unambiguous enough to generate working systems directly.** — confidence: high
  - Evidence: GitHub's spec-kit spec-driven.md states verbatim: 'Specifications don't serve code—code serves specifications'; 'The specification becomes the primary artifact. Code becomes its expression in a particular language and framework'; and 'Specifications must be precise, complete, and unambiguous enough to generate working systems.' Traditional specs were 'scaffolding we built and then discarded.' These are descriptive/definitional claims about what the methodology asserts, sourced to its authoritative primary doc. Caveat: spec-kit is GitHub's own advocacy document, so efficacy assertions in it would be marketing — but these claims report only the methodology's stated principles, not proven outcomes.
  - Sources: https://github.com/github/spec-kit/blob/main/spec-driven.md
- **At Google, design docs are informal documents capturing high-level implementation strategy and key design decisions with emphasis on trade-offs, written before coding to solve problems early rather than to produce code; they follow a typical structure (Context and Scope, Goals and Non-Goals, Actual Design, Alternatives Considered, Cross-Cutting Concerns like security/privacy/observability) and a four-phase lifecycle: creation/rapid iteration (mostly in Google Docs), review with a wider audience, implementation with doc updates as plans diverge, and maintenance where the doc serves as organizational memory.** — confidence: high
  - Evidence: Malte Ubl's industrialempathy.com article (the canonical account by an ex-Google engineer) states design docs are 'informal documents' that document 'the high level implementation strategy and key design decisions with emphasis on the trade-offs,' and that 'our job is not to produce code per se, but rather to solve problems' with unstructured text being 'the better tool for solving problems early.' It lists all five sections verbatim and describes the four-phase lifecycle, noting 'the vast majority of design docs at Google are created in Google Docs' and that engineers' first question on an unfamiliar system is 'Where is the design doc?' Caveat: the article frames the structure as informal guidelines, not a strict template.
  - Sources: https://www.industrialempathy.com/posts/design-docs-at-google/
- **RFCs originated as a document format with Steve Crocker's RFC 1 ('Host Software'), published April 7, 1969 as part of ARPANET, deliberately named 'requests for comments' to avoid a declarative tone and invite discussion — predating their later use in formal standards processes.** — confidence: high
  - Evidence: The RFC Editor's published RFC 1 bears author Steve Crocker (UCLA), title 'Host Software,' and date '7 April 1969.' Crocker wrote in RFC 1000: 'The basic ground rules were that anyone could say anything and that nothing was official. And to emphasize the point, I labeled the notes Request for Comments.' Wikipedia and Internet Society corroborate. Minor nuance: Crocker credits the concept to an early working-group discussion (Crocker, Carr, Rulifson) and is best described as having initiated/managed the series.
  - Sources: https://en.wikipedia.org/wiki/Request_for_Comments, https://www.rfc-editor.org/rfc/rfc1.html
- **The Rust RFC process provides a consistent, controlled path for substantial changes (language features, standard library additions) to enter the project so stakeholders are confident about its direction, requiring a design document from a template submitted as a pull request for community review and consensus BEFORE the change is implemented or merged.** — confidence: high
  - Evidence: The official Rust RFC process doc states the process provides 'a consistent and controlled path for new features to enter the language and standard libraries, so that all stakeholders can be confident about the direction.' It instructs authors to copy the template, fill in the RFC, submit a pull request for community design feedback and consensus, and notes that PRs implementing a feature without an RFC 'may be closed with a polite request to submit an RFC first'; only once an RFC is 'active' may authors implement it.
  - Sources: https://rust-lang.github.io/rfcs/0002-rfc-process.html
- **Oxide Computer uses RFDs (Requests for Discussion) as a document-driven design methodology to drive architectural and design decisions and document processes, APIs, and tools; its RFD workflow is derived from the Go proposal, Joyent RFD, Rust RFC, and Kubernetes proposal processes.** — confidence: high
  - Evidence: Oxide's blog states verbatim: 'At Oxide, RFDs (Requests for Discussion) play a crucial role in driving our architectural and design decisions. They document the processes, APIs, and tools that we use,' and 'The workflow for the RFD process is based upon those of the Golang proposal process, Joyent RFD process, Rust RFC process, and Kubernetes proposal process' — repeated in canonical RFD 1. Corroborated by a browsable public RFD corpus and open-source tooling (rfd-site, rfd-api), not merely marketing.
  - Sources: https://oxide.computer/blog/a-tool-for-discussion, https://rfd.shared.oxide.computer/rfd/0001
- **Michael Nygard introduced the Architecture Decision Record (ADR) in a November 15, 2011 blog post, proposing a defined structure (Title, Context, Decision, Status, Consequences) to document architecturally significant decisions, motivated by preserving rationale so future team members need not blindly accept or blindly change past decisions.** — confidence: high
  - Evidence: The Cognitect post (Nygard, Nov 15 2011) proposes documenting decisions 'that affect the structure, non-functional characteristics, dependencies, interfaces, or construction techniques,' lists the five sections in order, and frames the motivation: without rationale a new person 'has only two choices: blindly accept the decision or blindly change it.' Martin Fowler's bliki independently confirms Nygard 'coined the term Architecture Decision Record with an ADR-formatted article in 2011.' Minor nuance: decision logs predate ADRs (Kruchten), but the ADR concept specifically is Nygard's.
  - Sources: https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions, https://martinfowler.com/bliki/ArchitectureDecisionRecord.html
- **Amazon institutionalized narrative writing: Jeff Bezos mandated written narratives instead of PowerPoint in a 2004 S-Team email because narrative structure forces better thought, and the 'working backwards' PR-FAQ document combines a ~1-page press release written as if launching the product tomorrow with up to five pages of FAQs.** — confidence: high
  - Evidence: Bezos's June 9, 2004 email ('No powerpoint presentations from now on at steam') is documented by Amazon Chronicles, CNBC, and Inc., with his rationale: 'the narrative structure of a good memo forces better thought and better understanding of what's more important than what, and how things are related.' Dave Limp (then SVP of Devices) describes the working-backwards doc: 'The first page... is a press release, as if you were launching the product tomorrow. And then the next five pages are frequently asked questions.' The book 'Working Backwards' (Bryar/Carr) corroborates PR ~1 page + FAQ five pages or less. Note: the PR-FAQ is distinct from Amazon's separate six-pager narrative memo.
  - Sources: https://amazonchronicles.substack.com/p/working-backwards-dave-limp-on-amazons
- **RFCs / design documents are a convergent standardized engineering-planning practice across Big Tech companies and high-growth startups, with stated goals of org-wide information sharing, cross-team transparency about upcoming systems, and catching architectural problems before production; many firms settle on some form of the Design Document or RFC process.** — confidence: high
  - Evidence: Gergely Orosz (Pragmatic Engineer, ex-Uber) lists the RFC goals: 'Institute a strong, org-wide practice of information sharing... Provide transparency and good cross-team communication about upcoming systems... Catch potential architectural stumbling blocks before they get to production,' and states 'Many Big Tech companies and high-growth startups have ended up with a process similar to Uber's, and settle on a form of the Design Document or RFC process.' Companion sources name Google, Facebook, Microsoft, Amazon (PR-FAQ), Stripe, Airbnb. Claims are properly hedged as 'stated goals' and 'many companies.' Caveat: the 'similar to Uber's' framing overstates shared lineage — Google's informal design docs predate and differ structurally from Uber's RFC model.
  - Sources: https://newsletter.pragmaticengineer.com/p/rfcs-and-design-docs, https://www.industrialempathy.com/posts/design-docs-at-google/
- **The principal criticism of document/spec-first methodologies derives from the Big Design Up Front (BDUF) debate: BDUF completes and perfects design before implementation (associated with waterfall), and Agile advocates criticize it as poorly adaptable to changing requirements, as wrongly assuming designers can foresee problems without prototyping, and as risking analysis paralysis where planning cost exceeds the cost of fixing.** — confidence: high
  - Evidence: Wikipedia defines BDUF as 'a software development approach in which the program's design is to be completed and perfected before that program's implementation is started,' 'often associated with the waterfall model.' Agile critiques (attributed to agile practitioners): 'BDUF is poorly adaptable to changing requirements' and 'assumes that designers are able to foresee problem areas without extensive prototyping.' On analysis paralysis: 'If the cost of planning is greater than the cost of fixing then time spent planning is wasted.' Corroborated by Scott Ambler's BMUF essay and DevIQ. These are secondary-sourced but uncontroversial, well-documented characterizations of a stable methodological debate.
  - Sources: https://en.wikipedia.org/wiki/Big_design_up_front, https://agilemodeling.com

## Caveats
Source-quality and time-sensitivity notes: (1) The definitional claims for Documentation-Driven Development (zsup gist) and Spec-Driven Development (GitHub spec-kit) are sourced to the methodologies' own primary/advocacy documents. These are appropriate for descriptive 'what the methodology asserts' claims but are NOT independent evidence of efficacy — no verified claim here establishes that any of these methodologies actually improves outcomes, only what they prescribe. (2) 'DDD' is ambiguous: in common usage it denotes Domain-Driven Design (Eric Evans), not Documentation-Driven Development; the synthesis is scoped to the documentation/spec-driven sense per the research question. (3) Spec-Driven Development is recent (2024-2026) and tied to the AI-coding-agent wave; spec-kit is an actively evolving project, so specifics may shift. (4) Company-practice claims (Google design docs, Amazon memos/PR-FAQ, Oxide RFDs) describe internal practices via authoritative insider accounts but are point-in-time snapshots that may have evolved. (5) The BDUF criticism section relies on Wikipedia plus practitioner blogs (secondary, blog-quality) — adequate for an uncontested, stable debate but not primary research. (6) The 'companies converge on Uber's process' framing is somewhat Uber-centric and overstates a shared lineage that does not actually exist (Google's design-doc culture is independent and predates Uber's RFC model).

## Open questions
- Is there any empirical evidence (controlled studies or rigorous case data) that document-first methodologies improve software quality, velocity, or defect rates, as opposed to self-reported/advocacy claims? All verified claims are descriptive, none establish efficacy.
- How does Spec-Driven Development perform in practice with AI coding agents, and does the 'spec generates working systems' premise hold up beyond toy examples? The methodology is too new to have verified outcome evidence.
- What concrete failure modes and abandonment patterns occur for RFC/design-doc processes at scale (e.g., bureaucracy, endless discussion, power dynamics)? Critical sources like Jacob Kaplan-Moss were referenced as confirming adoption but their specific critiques were not among the verified claims.
- How do these document-driven approaches compare head-to-head with Test-Driven Development in measurable terms? The research surfaced that DDD sequences TDD after docs, but no verified comparative claim quantifies tradeoffs between the methodologies.
