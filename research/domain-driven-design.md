# domain driven design

> Generated by /batch-research → /deep-research on 2026-06-24

## Summary
Domain-Driven Design (DDD), a term coined by Eric Evans in his 2003 book "Domain-Driven Design: Tackling Complexity in the Heart of Software," is a software design methodology built on the premise that the heart of software is its ability to solve the user's domain-related problems, with all other features supporting that purpose; it therefore models software to match a business domain based on input from domain experts, aligning code structure and language to that domain. DDD rests on three pillars — the Ubiquitous Language (a shared vocabulary co-created by developers and domain experts and used relentlessly in both communication and code), strategic design (large-scale structure, dominated by the Bounded Context pattern that delimits where a model applies and divides large systems instead of forcing a single unified model), and tactical design (patterns such as entities, value objects, aggregates/aggregate roots, repositories, domain events, factories, and domain services for building the model). Vaughn Vernon's "Implementing Domain-Driven Design" (the "Red Book") and "Domain-Driven Design Distilled" codified the canonical aggregate design rules: model true invariants in a single transactional consistency boundary with exactly one root, design small aggregates, reference other aggregates by identity only, and use eventual consistency (via domain events) across aggregate boundaries. DDD maps naturally onto microservices and event-driven architecture — each bounded context is a microservice candidate, with a service ideally no smaller than an aggregate and no larger than a bounded context, and cross-aggregate consistency achieved asynchronously through domain events. The methodology's principal failure mode is the anemic domain model (business logic placed in service classes rather than entities), an antipattern that undermines DDD's core benefit, while its costs and over-engineering risk define where it is and isn't appropriate.

(2) Several claims about tactical/aggregate patterns are anchored on Microsoft Learn's Azure Architecture Center, a credible but secondary vendor source; however, each restates canonical Vernon (IDDD) rules independently confirmed via the InformIT publisher excerpts (primary). (3) The microservices/bounded-context sizing heuristic (Finding 12) is the most contested item — practitioners disagree on whether a bounded context equals, exceeds, or is decomposed into microservices (Khononov dissents on the upper bound); treat it as a debated design heuristic, not settled fact. (4) The anemic-domain-model antipattern (Finding 7) has a documented minority dissent (FP/SOLID advocates) arguing it is acceptable outside the DDD frame; the claim is scoped to 'within DDD,' which holds. (5) Only one source claim (the framing 'DDD is primarily about modeling a Ubiquitous Language in an explicitly Bounded Context') drew a non-unanimous 2-1 vote; it is complementary to, not contradictory with, Evans' 'domain model at the heart of software' framing. (6) DDD is a stable, foundational methodology (Evans 2003), so datedness is low-risk for definitional claims; the Microsoft architecture pages are current (updated early 2026).
